Docker简介

Docker 是一个开源的应用容器引擎，基于 Go 语言，并遵从 Apache2.0 协议开源。它可以让开发者打包应用以及依赖包到一个轻量级、可移植的容器中，然后发布到任何流行的 Linux 机器上，也可以实现虚拟化。Docker 可用于开发应用、交付应用、运行应用等场景。
容器是完全使用沙箱机制，相互之间不会有任何接口，更重要的是容器性能开销极低。Docker 允许用户将基础设施中的应用单独分割出来，形成更小的部分容器，从而提高软件交付速度。
Docker 从 17.03 版本之后分为 CE（Community Edition: 社区版）和 EE（Enterprise Edition: 企业版），通常使用社区版就可以满足大多数业务场景。

Docker理念

Docker的主要目标是“Build，Ship and Run Any App,Anywhere”，也就是通过对应用组件的封装、分发、部署、运行等生命周期的管理，使用户的APP（可以是一个WEB应用或数据库应用等等）及其运行环境能够做到“一次镜像，处处运行”。

Linux容器技术的出现就解决了这样一个问题，而 Docker 就是在它的基础上发展过来的。将应用打成镜像，通过镜像成为运行在Docker容器上面的实例，而 Docker容器在任何操作系统上都是一致的，这就实现了跨平台、跨服务器。只需要一次配置好环境，换到别的机子上就可以一键部署好，大大简化了操作。

Docker 容器与虚拟机的区别

Docker 容器与虚拟机很相似，但它们之间又有着很大的区别。
虚拟机是通过软件模拟的具有完整硬件系统功能的、运行在一个完全隔离环境中的完整计算机系统。创建虚拟机时，会将实体机的部分硬盘和内存容量作为虚拟机的硬盘和内存，每个虚拟机都有独立的硬盘和操作系统，可以像使用实体机一样对虚拟机进行操作。
虚拟机会消耗大量系统资源和开销，尤其是当多个虚拟机在同一物理服务器上运行时，每个虚拟机都有自己的子操作系统，大量精力以及资源被用于虚拟化的部署和运行上。
容器类似于虚拟机，只是容器不是完整的操作系统，容器通常只包含必要的操作系统包和应用程序，这就是它们轻量级的原因。

镜像命令

docker images

各个选项说明:
REPOSITORY：表示镜像的仓库源
TAG：镜像的标签版本号
IMAGE ID：镜像ID
CREATED：镜像创建时间
SIZE：镜像大小

同一仓库源可以有多个 TAG版本，代表这个仓库源的不同个版本，我们使用 REPOSITORY:TAG 来定义不同的镜像。
如果你不指定一个镜像的版本标签，例如你只使用 ubuntu，docker 将默认使用 ubuntu:latest 镜像

• OPTIONS说明：
• -a :列出本地所有的镜像（含历史映像层）
• -q :只显示镜像ID。

心得体会

在学习本地镜像发布到阿里云的过程中，我发现以下几点体会：
首先，我们需要了解 Docker 镜像的构建和发布流程。通常来说，我们需要先在本地构建出需要发布的镜像，这个过程可以通过 Dockerfile 来实现，然后通过 Docker CLI 发布到 Docker Hub 或者其他 Docker Registry 上。在这个流程中，我们需要注意镜像的命名和标签，这些信息会决定镜像在 Registry 中的位置和版本。
针对阿里云的镜像服务，我们可以通过创建自己的 Registry 实例来发布镜像。在创建 Registry 实例的过程中，我们需要确定镜像存储的地域和网络类型，并设置相应的访问控制策略。此外，我们还需要获取到自己的 Access Key 和 Secret Key，用来进行认证和授权操作。
在本地构建完镜像之后，我们可以通过 Docker CLI 的 push 命令将镜像上传到阿里云的 Registry 上。在上传镜像的过程中，我们需要注意镜像的命名和标签，确保与之前设置的信息一致，同时还需要进行认证和授权操作，以确保上传操作的安全性和正确性。
最后，在使用上传的镜像时，我们需要注意镜像版本和存储位置。在阿里云的镜像服务中，我们可以通过镜像名称和标签来确定需要使用的镜像版本，同时还需要确定镜像存储的地域和网络类型，以确保能够正常访问和使用镜像。
通过以上几点体会，我认为学习本地镜像发布到阿里云可以帮助我们更好地理解和掌握 Docker 镜像的构建和发布技术，同时也可以帮助我们更好地使用阿里云的镜像服务，提高我们的工作效率和技术水平
